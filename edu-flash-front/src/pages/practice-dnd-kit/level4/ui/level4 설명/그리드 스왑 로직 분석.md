# Level 4: 1:1 그리드 스왑 (Swapping) 완벽 분석

이전 레벨에서는 아이템들이 비켜주는 "밀어내기(Reordering)" 방식이었다면, 레벨 4는 체스 말의 위치를 바꾸듯 정확히 **두 아이템의 자리만 맞바꾸는 "양자 교체(Swapping)"** 방식을 구현합니다.

---

## 1. 왜 SortableContext를 안 쓰는가?

`@dnd-kit/sortable`의 `SortableContext`는 근본적으로 **"밀어내기"** 를 위해 설계되었습니다.
- 1번을 5번 위치로 드래그하면 → 2, 3, 4번이 한 칸씩 밀림
- 드래그 중에 다른 아이템들이 "비켜주는" 애니메이션이 자동 적용
- 이건 리스트에서는 자연스럽지만, **그리드에서 1:1 교체**와는 맞지 않음

따라서 레벨 1처럼 기초적인 **`useDraggable` + `useDroppable`만 사용**하고, swap 로직과 애니메이션은 직접 구현합니다.

---

## 2. 삽질의 과정: 왜 framer-motion이 필요한가

### 시도 1: SortableContext + onDragEnd에서 swap
```tsx
// ❌ 문제: 드래그 중에는 SortableContext의 "밀림" 프리뷰가 보임
// 드롭 시에만 swap해도 드래그 중 UX가 밀어내기라 어색함
```

### 시도 2: useDraggable + useDroppable + CSS pop 애니메이션
```tsx
// ❌ 문제: 드롭 순간 state가 바뀌면서 한 프레임에 리렌더링
// CSS animation으로 pop 효과를 줘도 "마지막 동작만 부드러움"
// 교체되는 과정 자체가 보이지 않음 (텔레포트)
```

### 시도 3: useDraggable + framer-motion layoutId + onDragOver 미리보기 swap
```tsx
// ❌ 문제: useDraggable의 CSS transform과 framer-motion의 layout 애니메이션이 충돌
// 드래그 중 요소가 떨리거나 위치가 꼬임
// 두 라이브러리가 동시에 같은 요소의 위치를 제어하려고 싸움
```

### 최종: useDraggable + framer-motion `layout` + onDragEnd에서만 swap ✅
```tsx
// ✅ 드래그 중: DragOverlay가 마우스를 따라다님 + 드롭 대상 하이라이트
// ✅ 드롭 시: 배열 swap → framer-motion layout이 위치 변화 감지 → 스르르 애니메이션
// 핵심: 드래그 중에는 framer-motion을 건드리지 않음 (충돌 회피)
```

## 3. 최종 코드 구조 분석

### 3-1. GridCell 컴포넌트 — 하나의 요소가 3가지 역할

```tsx
function GridCell({ item, activeId }) {
  // 역할 1: 드래그 시작 감지 (transform은 적용 안 함!)
  const { attributes, listeners, setNodeRef: setDragRef, isDragging } = useDraggable({ id: item.id })

  // 역할 2: 드롭 대상 감지
  const { setNodeRef: setDropRef, isOver } = useDroppable({ id: item.id })

  return (
    // 역할 3: 위치 변경 시 애니메이션 (framer-motion)
    <motion.div
      layout  // ← 이것만으로 위치 바뀔 때 자동 애니메이션
      transition={{ type: 'spring', stiffness: 400, damping: 28 }}
      ref={(node) => { setDragRef(node); setDropRef(node) }}  // 두 ref를 하나의 DOM에
      {...attributes}
      {...listeners}
    >
      {item.label}
    </motion.div>
  )
}
```

**핵심:** `useDraggable`의 `transform`을 style에 적용하지 않습니다!
- 원본은 제자리에 있되 `opacity-0`으로 숨김
- DragOverlay가 마우스를 따라다니는 복제본 역할

### 3-2. handleDragEnd — 드롭 시에만 swap

```tsx
const handleDragEnd = (event: DragEndEvent) => {
  const { active, over } = event
  setActiveId(null)

  if (over && active.id !== over.id) {
    setItems((prev) => {
      const arr = [...prev]
      const oldIndex = arr.findIndex((i) => i.id === active.id)
      const newIndex = arr.findIndex((i) => i.id === over.id)
      // 양자 교체: 두 요소만 바꾸고 나머지는 그대로
      ;[arr[oldIndex], arr[newIndex]] = [arr[newIndex], arr[oldIndex]]
      return arr
    })
  }
}
```

### 3-3. key={item.id}가 중요한 이유

```tsx
{items.map((item) => (
  <GridCell key={item.id} item={item} activeId={activeId} />
))}
```

- `key={item.id}` → React가 각 아이템을 ID로 추적
- 배열 순서가 바뀌면 React는 DOM을 재배치
- `motion.div layout`이 이 재배치를 감지 → 스프링 애니메이션 적용
- 만약 `key={index}`였다면? React는 같은 위치의 props만 바꿔서 애니메이션 안 됨!

### 3-4. DragOverlay — 마우스를 따라다니는 분신

```tsx
<DragOverlay dropAnimation={null}>
  {activeItem ? (
    <div className={`... shadow-lg ${activeItem.color}`}>
      {activeItem.label}
    </div>
  ) : null}
</DragOverlay>
```

- `dropAnimation={null}` → 드롭 시 DragOverlay가 원래 위치로 돌아가는 애니메이션 제거
- 이걸 안 하면 드롭 후 DragOverlay가 스르르 사라지면서 깜빡임 발생

## 4. 요약: 레벨 3 vs 레벨 4

| 항목 | Level 3 (밀어내기) | Level 4 (양자 교체) |
| :--- | :--- | :--- |
| **방식** | 새치기 (나머지 밀림) | 1:1 맞바꾸기 (나머지 가만히) |
| **dnd-kit** | `useSortable` + `SortableContext` | `useDraggable` + `useDroppable` |
| **핵심 메서드** | `arrayMove` | `[a, b] = [b, a]` |
| **애니메이션** | dnd-kit 내장 | `framer-motion` `layout` prop |
| **드래그 중** | 실시간 자리 비킴 | 하이라이트만 |
| **드롭 시** | 이미 정렬됨 | swap + 스프링 애니메이션 |
| **적합한 UI** | 세로 리스트, 순위 | 격자(Grid), 고정 자리 |

## 5. 교훈

- **dnd-kit의 transform과 framer-motion의 layout은 같은 요소에서 충돌한다**
- 드래그 중 미리보기 swap은 두 라이브러리가 동시에 위치를 제어하려 해서 깨진다
- 해결: **역할 분리** — dnd-kit은 드래그/드롭 로직만, framer-motion은 애니메이션만
- `key={item.id}`를 빠뜨리면 모든 layout 애니메이션이 죽는다
