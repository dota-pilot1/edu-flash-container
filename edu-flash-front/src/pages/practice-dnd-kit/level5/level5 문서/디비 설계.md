# Level 5 — 트리 구조 DB 설계 가이드

계층형 구조(트리)를 관계형 데이터베이스(RDB)에 저장할 때 가장 표준적으로 사용되는 **인접 리스트(Adjacency List)** 모델을 기반으로 설계합니다.

## 1. 테이블 스키마 (Table Schema)

| 컬럼명 | 타입 | 설명 | 비고 |
| :--- | :--- | :--- | :--- |
| **id** | VARCHAR / INT | 고유 식별자 (ID) | Primary Key |
| **label** | VARCHAR | 메뉴 또는 아이템의 이름 | |
| **parent_id** | VARCHAR / INT | 부모 노드의 ID | 최상위 노드는 NULL (Foreign Key) |
| **sort_order** | INT | 동일 레벨(부모) 내의 정렬 순서 | DnD 정렬 시 핵심 컬럼 |
| **depth** | INT | 노드의 깊이 정보 (1차, 2차...) | 렌더링 및 필터링 시 활용 |

## 2. 왜 이 방식을 사용하나요? (Pros)
1. **표준적 설계**: 가장 널리 쓰이는 방식으로, 어떤 DB에서도 쉽게 구현 가능합니다.
2. **이동(Move) 용이성**: `parent_id` 값 하나만 바꾸면 서브트리 전체의 위치를 옮길 수 있습니다.
3. **간결한 CRUD**: 데이터의 삽입(Insert)과 삭제(Delete) 로직이 단순합니다.

## 3. 드래그 앤 드롭(DnD) 시 데이터 업데이트 로직

사용자가 화면에서 순서를 바꿀 경우, 프론트엔드에서 변경된 **플랫 배열(Flat Array)**의 상태를 서버로 전달하고 서버는 이를 DB에 반영합니다.

### 정렬 순서 업데이트 방식
1. **일괄 업데이트 (Bulk Update)**: 해당 depth의 모든 아이템 ID 리스트를 서버에 보내고, DB에서 `sort_order`를 상위부터 1, 2, 3... 순서대로 다시 업데이트합니다. (가장 구현이 쉬움)
2. **개별 업데이트**: 바뀐 노드의 `sort_order`만 업데이트하고 나머지 노드들의 index를 연산하여 조정합니다.

## 4. 데이터 예시 (SQL 기준)

| id | label | parent_id | sort_order | depth |
| :--- | :--- | :--- | :--- | :--- |
| menu-1 | 홈 | NULL | 1 | 1 |
| menu-2 | 학습 | NULL | 2 | 1 |
| menu-2-1 | 프론트엔드 | menu-2 | 1 | 2 |
| menu-2-2 | 백엔드 | menu-2 | 2 | 2 |
| menu-3 | 내 카드 | NULL | 3 | 1 |

## 5. 조회(Read) 성능 최적화
- **인덱스(Index)**: `parent_id`와 `sort_order` 컬럼에 인덱스를 걸어두면 트리를 구성하기 위한 데이터를 가져올 때 속도가 매우 빨라집니다.
- **재귀 쿼리**: SQL의 `WITH RECURSIVE` 문을 사용하면 단 한 번의 쿼리로 전체 계층 구조를 계층 순서대로 가져올 수 있습니다.
